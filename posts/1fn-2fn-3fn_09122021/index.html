1NF, 2NF, and 3NF Normalization

September 12, 2021

Database normalization is a crucial step in designing database systems.

Intro

Normalization improves data integrity and reduces data redundancy. In this tutorial we will go over the common types of normalization - First Normal form (1NF), Second Normal form (2NF), and Third Normal form (3NF).

Initial data

We have a dataset of baseball games that shows which players scored runs.

Game ID Player Names Player IDs Date Zip Code City State
1 Mark, Thomas, Scott 1,2,3 1/01/2021 60601 Chicago IL
2 Mark 1 1/08/2021 53794 Madison WI

The data contains 2 games noted by the Game ID. Mark, Thomas, and Scott scored in game 1 while Mark scored in game 2. Each player also has a corresponding id - Mark’s id is 1 and Thomas’s id is 2. The Date column specifies when a game took place. The Zip Code, City, and State columns specify where a game took place.

1NF

1st Normal Form critiera:

  • Each cell must contain a single value

In our table the Player Names and Player IDs column contain multiple values. The first row has Mark, Thomas, and Scott in the Player Names column. We can satisfy the 1NF criteria by creating a separate row for each player.

Game ID Player Names Player IDs Date Zip Code City State
1 Mark 1 1/01/2021 60601 Chicago IL
1 Thomas 2 1/01/2021 60601 Chicago IL
1 Scott 3 1/01/2021 60601 Chicago IL
2 Mark 1 1/08/2021 53794 Madison WI

The data now meets 1NF! Each cell contains one value.

Note that now each row in the table is unique by the combination of Game ID and Player ID. We can say that these 2 columns form a composite primary key.

2NF

2nd Normal Form criteria:

  • meet 1NF criteria
  • should not contain any partial dependencies

The textbook definition of a partial dependency is when “non-prime attribute (attributes which are not part of any candidate key) is dependent on any proper subset of any candidate key of the table.”

Let’s break it down:

  • “non prime attributes” are columns that do not belong to the composite or primary key.
    • The composite key for our data is Game ID and Player ID
    • The “non prime attributes” for our data are Date, City, State and Player Name
  • Do these columns depend on the composite key? If so, we violate 2NF.
    • Date column is dependent on Game ID. Game ID is part of the composite key thus we violate 2NF
    • Zip Code column is dependent on Game ID. Game ID is part of the composite key thus we violate 2NF
    • City column is dependent on Game ID. Game ID is part of the composite key thus we violate 2NF
    • State column is dependent on Game ID. Game ID is part of the composite key thus we violate 2NF
    • Player Name column is dependent on Player ID. Player ID is part of the composite key thus we violate 2NF

We can satisfy 2NF criteria by splitting the table into smaller tables:

Game table: Store all game specific information in one table.

Game ID Date Zip Code City State
1 1/01/2021 60601 Chicago IL
1 1/01/2021 60601 Chicago IL
1 1/01/2021 60601 Chicago IL
2 1/01/2021 53794 Madison WI

Player table: Store all player specific information in one table

Player ID Player Name
1 Mark
2 Thomas
3 Scott

Scores table: Store when each player scores in a game

Game ID Player ID
1 1
1 2
1 3
2 1

The tables are now in 2NF. The composite key consisted of Game ID and Player ID. Now all columns that are dependent on Game ID are in a separate Game table and all columns that are dependent on Player ID are in a separate Player table.

3NF

3rd Normal Form criteria:

  • Must be in 2NF
  • No transitive dependeny for non prime attributes.

What is a transitive dependency?

If B depends on A and C depends on A, then C transitively depends on A.

Let’s look at our Game table:

  • City depends on Zip Code -> Zip Code depends on Game ID -> City transitively depends on Game ID
  • State depends on Zip Code -> Zip Code depends on Game ID -> State transitively depends on Game ID

We need to remove these transitive dependencies from the Game table to put it in 3NF.

One approach is to remove the City and State columns. We can understand the city and state based on the zip code, so why store the extra data?

Game table:

Game ID Date Zip Code
1 1/01/2021 60601
1 1/01/2021 60601
1 1/01/2021 60601
2 1/01/2021 53794

However if we want to store the city and state information, we can create a separate lookup table for location information:

Location reference table:

Zip Code City State
60601 Chicago IL
53794 Madison WI

All of our data is now in 3NF!

Conclusion

You are now able to transform data to 1NF, 2NF, and 3NF.
